# Change Lifecycle

## Overview

Manage the full change request lifecycle — from creation through approval, implementation, and closure.

**When to use:**
- "Create a change request for this deployment"
- "What changes are pending approval?"
- "Approve this change request"
- "Add implementation tasks to this change"

## Prerequisites

- **Roles:** `itil` for viewing; `change_manager` or `approver_user` for approvals
- **MCP Package:** `platform_developer` or higher

## Procedure

### Step 1: Create Change Request

```
Tool: create_change_request
Parameters:
  short_description: "Deploy benefits widget company code update"
  description: "Add 2J00 company code branch to benefits widget client script. Tested in dev, ready for production."
  type: "normal"
  category: "Software"
  risk: "moderate"
  impact: "2"
  confirm: "approve"
```

### Step 2: Add Implementation Tasks

```
Tool: add_change_task
Parameters:
  change_id: "<change sys_id>"
  short_description: "Update sp_widget client_script"
  description: "Apply the code patch generated by widget-patching skill"
  assignment_group: "Portal Development"
  confirm: "approve"
```

```
Tool: add_change_task
Parameters:
  change_id: "<change sys_id>"
  short_description: "Validate in staging"
  description: "Run portal-diagnosis skill against staging instance"
  assignment_group: "QA Team"
  confirm: "approve"
```

### Step 3: Submit for Approval

```
Tool: submit_change_for_approval
Parameters:
  change_id: "<change sys_id>"
  confirm: "approve"
```

### Step 4: Check Approval Status

```
Tool: get_change_request_details
Parameters:
  change_id: "<change sys_id>"
```

### Step 5: Approve or Reject

```
Tool: approve_change
Parameters:
  change_id: "<change sys_id>"
  approval_comments: "Reviewed code diff. Change is low-risk and well-tested."
  confirm: "approve"
```

Or reject:
```
Tool: reject_change
Parameters:
  change_id: "<change sys_id>"
  rejection_reason: "Missing rollback plan. Please add a snapshot reference."
  confirm: "approve"
```

### Step 6: Track All Changes

```
Tool: list_change_requests
Parameters:
  state: "implement"
  limit: 20
```

## Tips

- Combine with `changeset-workflow`: create change request → make changes → commit changeset → link to change
- Combine with `widget-patching`: create snapshot → reference snapshot path in change task as rollback plan
- One change request per logical deployment — don't mix unrelated changes
