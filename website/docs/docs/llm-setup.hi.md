# MFA ServiceNow MCP — LLM सेटअप गाइड

> **यह फ़ाइल LLM एजेंट्स के लिए है, मनुष्यों के लिए नहीं।**
> एक मनुष्य नीचे दिए गए curl कमांड को अपने AI कोडिंग असिस्टेंट में पेस्ट करता है।
> LLM इस फ़ाइल को पढ़ता है और उपयोगकर्ता को इंटरैक्टिव रूप से सेटअप के माध्यम से ले जाता है।

---

## उपयोग कैसे करें (मनुष्यों के लिए)

इसे अपने AI कोडिंग असिस्टेंट (Claude Code, Cursor, Codex, OpenCode, Windsurf, आदि) में पेस्ट करें:

```
Install and configure mfa-servicenow-mcp by following:
curl -s https://raw.githubusercontent.com/jshsakura/mfa-servicenow-mcp/main/docs/llm-setup.md
```

बस इतना ही। बाकी काम आपका AI संभाल लेता है।

---

## LLM एजेंट के लिए निर्देश

आप **mfa-servicenow-mcp** सेट कर रहे हैं — एक MCP सर्वर जो AI एजेंट्स को MFA/SSO समर्थन के साथ ServiceNow प्लेटफ़ॉर्म तक पहुँच देता है। नीचे दिए गए हर चरण का क्रम में पालन करें। उपयोगकर्ता से इंटरैक्टिव रूप से प्रश्न पूछें; मान न मानें।

### Step 0 — पर्यावरण का पता लगाएँ

1. ऑपरेटिंग सिस्टम का पता लगाएँ: `uname -s 2>/dev/null || echo Windows` चलाएँ
2. पता लगाएँ कि आप किस AI क्लाइंट के अंदर चल रहे हैं। इस मैपिंग का उपयोग करें:

| संकेत | क्लाइंट |
|--------|--------|
| आपके पास `claude mcp add` या `.claude/` डायरेक्टरी तक पहुँच है | **claude-code** |
| आप Cursor IDE के अंदर हैं या `.cursor/` मौजूद है | **cursor** |
| आपके पास OpenCode CLI तक पहुँच है या `opencode.json` मौजूद है | **opencode** |
| आप Codex CLI के अंदर हैं या `.codex/` मौजूद है | **codex** |
| आप Windsurf IDE के अंदर हैं या `.windsurf/` मौजूद है | **windsurf** |
| आप Copilot के साथ VS Code के अंदर हैं | **vscode-copilot** |
| आप Antigravity के अंदर हैं | **antigravity** |
| आप Zed एडिटर के अंदर हैं या `~/.config/zed/` मौजूद है | **zed** |
| उपरोक्त में से कोई नहीं | उपयोगकर्ता से पूछें कि वे किस क्लाइंट का उपयोग करते हैं |

3. यदि आप स्वतः पता नहीं लगा सकते, तो पूछें:
   > आप कौन सा AI कोडिंग टूल उपयोग कर रहे हैं?
   > 1. Claude Code
   > 2. Claude Desktop
   > 3. Cursor
   > 4. OpenCode
   > 5. Codex (OpenAI)
   > 6. Windsurf
   > 7. VS Code Copilot
   > 8. Zed
   > 9. AntiGravity (Google)

परिणाम को `$CLIENT` के रूप में संग्रहीत करें।

### Step 1 — uv इंस्टॉल करें

जाँचें कि क्या `uv` पहले से इंस्टॉल है: `uv --version`

यदि इंस्टॉल नहीं है:

- **macOS / Linux:**
  ```bash
  curl -LsSf https://astral.sh/uv/install.sh | sh
  ```
- **Windows (PowerShell):**
  ```powershell
  powershell -ExecutionPolicy ByPass -c "irm https://astral.sh/uv/install.ps1 | iex"
  ```

इंस्टॉल के बाद, सत्यापित करें: `uv --version`
यदि कमांड नहीं मिलती, तो उपयोगकर्ता को अपना शेल पुनः आरंभ करना पड़ सकता है या PATH में `~/.local/bin` जोड़ना पड़ सकता है।

### Step 2 — Playwright Chromium इंस्टॉल करें (अनिवार्य, छोड़ें नहीं)

> हार्ड निर्भरता। इसे छोड़ना ही फ़ील्ड में सेटअप विफल होने का #1 कारण है।
> यह न मानें कि यह पहले से इंस्टॉल है। उपयोगकर्ता को इसे टालने न दें।
> जब तक यह सफल न हो जाए, Step 3 पर आगे न बढ़ें।

**2.1 — जाँचें कि क्या Chromium पहले से इंस्टॉल है**

- macOS: `ls ~/Library/Caches/ms-playwright/chromium-*/chrome-mac/Chromium.app/Contents/MacOS/Chromium 2>/dev/null`
- Linux: `ls ~/.cache/ms-playwright/chromium-*/chrome-linux/chrome 2>/dev/null`
- Windows (PowerShell): `Get-ChildItem "$env:USERPROFILE\AppData\Local\ms-playwright\chromium-*\chrome-win\chrome.exe" -ErrorAction SilentlyContinue`

यदि कोई पथ प्रिंट होता है, तो Chromium पहले से इंस्टॉल है — Step 3 पर जाएँ।

**2.2 — Chromium इंस्टॉल करें**

यदि 2.1 में कुछ नहीं मिला, तो `uvx` के माध्यम से Chromium इंस्टॉल करें ताकि Playwright सेटअप MCP सर्वर के समान ही निष्पादन शैली का उपयोग करे:

```bash
uvx --with playwright playwright install chromium
```

यह पहली बार ~150 MB डाउनलोड करता है। धीमे कनेक्शन पर इसमें कई मिनट लग सकते हैं — यह सामान्य है। जल्दी अबॉर्ट न करें। उपयोगकर्ता को एक प्रगति संदेश दिखाएँ ("ServiceNow MFA लॉगिन के लिए Chromium डाउनलोड हो रहा है — धीमे नेटवर्क पर इसमें कुछ मिनट लग सकते हैं…") ताकि वे प्रतीक्षा को समझ सकें।

यदि `uvx` पैकेज निष्पादन अवरुद्ध है, तो रिलीज़ zip/exe पथ पर स्विच करें:

- GitHub Releases से `servicenow-mcp-<platform>-<version>.zip` डाउनलोड करें। कोई इंस्टॉलर स्क्रिप्ट नहीं है — zip में केवल PyInstaller-निर्मित एक्ज़ीक्यूटेबल होता है।
- एक्ज़ीक्यूटेबल को उपयोगकर्ता के नियंत्रण वाले किसी स्थिर फ़ोल्डर में एक्सट्रैक्ट करें (जैसे `~/apps/servicenow-mcp/`)।
- यदि ब्राउज़र डाउनलोड भी अवरुद्ध है, तो उसी रिलीज़ से `ms-playwright-chromium-<platform>-<version>.zip` डाउनलोड करें और इसे `ms-playwright/` नामक एक सिबलिंग फ़ोल्डर में एक्सट्रैक्ट करें — एक्ज़ीक्यूटेबल स्टार्टअप पर उस लेआउट का स्वतः पता लगाता है और अपनी प्रक्रिया के लिए `PLAYWRIGHT_BROWSERS_PATH` को उस पर सेट करता है।
- MCP क्लाइंट के `command` को उस एक्ज़ीक्यूटेबल के पूर्ण पथ पर सेट करें। env ब्लॉक uvx सेटअप के समान है।

**2.3 — सत्यापित करें और विफलता पर रुकें**

2.1 से जाँच पुनः चलाएँ। यदि बाइनरी अभी भी अनुपस्थित है, तो **सेटअप रोकें** और सटीक कमांड आउटपुट के साथ उपयोगकर्ता को विफलता की रिपोर्ट करें। सामान्य कारण:

- कॉर्पोरेट नीति जो पैकेज या ब्राउज़र डाउनलोड को अवरुद्ध करती है; रिलीज़ zip/exe पथ का उपयोग करें
- एंटीवायरस जो Chromium आर्काइव को क्वारंटाइन कर रहा है
- डिस्क भर गई है

Chromium अनुपस्थित होने पर Step 3 पर आगे **न बढ़ें**। MCP सर्वर शुरू होता हुआ दिखाई देगा, पहला टूल कॉल हैंग हो जाएगा, और उपयोगकर्ता की लॉगिन विंडो कभी नहीं खुलेगी — ठीक वही विफलता मोड जिसे रोकने के लिए यह चरण मौजूद है।

**यह क्यों मायने रखता है (एजेंट के लिए संदर्भ — जब तक पूछा न जाए, उपयोगकर्ता को न दिखाएँ):** जब Chromium अनुपस्थित होता है, तो रनटाइम "ऑन डिमांड इंस्टॉल" पर फ़ॉलबैक करने की कोशिश करता है, लेकिन धीमे कनेक्शन पर वह डाउनलोड MCP होस्ट के टूल-कॉल टाइमआउट से अधिक हो जाता है। उपयोगकर्ता को कोई लॉगिन विंडो नहीं दिखती, कोई एरर UI नहीं दिखता, और वे मान लेते हैं कि सर्वर खराब है। यहाँ पहले से इंस्टॉल करने से पहला टूल कॉल सबसेकंड हो जाता है।

### Step 3 — उपयोगकर्ता कॉन्फ़िगरेशन एकत्र करें

उपयोगकर्ता से ये प्रश्न एक-एक करके पूछें। ब्रैकेट में डिफ़ॉल्ट प्रदान करें।

1. **ServiceNow इंस्टेंस URL**
   > आपका ServiceNow इंस्टेंस URL क्या है?
   > उदाहरण: `https://your-company.service-now.com`

   `$INSTANCE_URL` के रूप में संग्रहीत करें। सत्यापित करें कि यह एक URL जैसा दिखता है।

2. **प्रमाणीकरण प्रकार**
   > आप ServiceNow में कैसे प्रमाणीकरण करते हैं?
   > 1. browser — वास्तविक ब्राउज़र के माध्यम से MFA/SSO (अनुशंसित)
   > 2. basic — उपयोगकर्ता नाम + पासवर्ड
   > 3. oauth — OAuth 2.0 क्लाइंट क्रेडेंशियल
   > 4. api_key — REST API कुंजी

   `$AUTH_TYPE` के रूप में संग्रहीत करें। डिफ़ॉल्ट: `browser`

3. **क्रेडेंशियल** (वैकल्पिक, ब्राउज़र auth के साथ फ़ॉर्म पूर्व-भरण के लिए)
   > (वैकल्पिक) लॉगिन फ़ॉर्म को पूर्व-भरण करने के लिए अपना ServiceNow उपयोगकर्ता नाम दर्ज करें।
   > हर बार इसे मैन्युअल रूप से टाइप करने के लिए खाली छोड़ दें।

   `$USERNAME` के रूप में संग्रहीत करें (खाली हो सकता है)।
   यदि प्रदान किया गया है, तो `$PASSWORD` भी पूछें।

4. **टूल पैकेज**
   > आपको किस टूल पैकेज की आवश्यकता है?
   > 1. standard — कोर टूल्स (incidents, changes, catalog) [डिफ़ॉल्ट]
   > 2. service_desk — Standard + असाइनमेंट, SLA, एस्केलेशन
   > 3. portal_developer — Standard + पोर्टल विजेट, पेज, थीम
   > 4. platform_developer — Standard + स्क्रिप्ट, flows, update sets
   > 5. full — बंडल किए गए वर्कफ़्लो के साथ सबसे व्यापक पैकेज सतह (53 टूल्स)

   `$TOOL_PACKAGE` के रूप में संग्रहीत करें। डिफ़ॉल्ट: `standard`

5. **हेडलेस ब्राउज़र**
   > ब्राउज़र को हेडलेस मोड में चलाएँ? (कोई दृश्यमान विंडो नहीं)
   > अनुशंसित: नहीं (ताकि आप MFA प्रॉम्प्ट देख सकें और पूरा कर सकें)

   `$HEADLESS` के रूप में संग्रहीत करें। डिफ़ॉल्ट: `false`

### Step 4 — इंस्टॉलर कमांड चलाएँ

**महत्वपूर्ण: जब क्लाइंट समर्थन करता है, तो हमेशा प्रोजेक्ट-लोकल इंस्टॉलेशन को डिफ़ॉल्ट करें।** `--scope global` का उपयोग केवल तभी करें जब उपयोगकर्ता स्पष्ट रूप से ग्लोबल इंस्टॉल माँगता है।

एक एकल इंस्टॉलर कमांड बनाएँ और इसे वर्तमान प्रोजेक्ट रूट से चलाएँ। इंस्टॉलर अब इनका मालिक है:
- क्लाइंट-विशिष्ट कॉन्फ़िग फ़ाइल पथ
- मौजूदा कॉन्फ़िग फ़ाइलों के लिए मर्ज/अपडेट व्यवहार
- समर्थित क्लाइंट्स के लिए वैकल्पिक स्किल्स इंस्टॉलेशन

बेस कमांड:

```bash
uvx --with playwright --from mfa-servicenow-mcp servicenow-mcp setup "$CLIENT" \
  --instance-url "$INSTANCE_URL" \
  --auth-type "$AUTH_TYPE" \
  --tool-package "$TOOL_PACKAGE" \
  --browser-headless "$HEADLESS"
```

केवल आवश्यकता होने पर फ़्लैग जोड़ें:

- यदि उपयोगकर्ता ने उपयोगकर्ता नाम प्रदान किया: `--username "$USERNAME"`
- यदि उपयोगकर्ता ने पासवर्ड प्रदान किया: `--password "$PASSWORD"`
- OAuth के लिए: `--client-id`, `--client-secret`, और वैकल्पिक रूप से `--token-url` जोड़ें
- API कुंजी के लिए: `--api-key` और वैकल्पिक रूप से `--api-key-header` जोड़ें
- यदि उपयोगकर्ता ग्लोबल इंस्टॉल चाहता है: `--scope global` जोड़ें
- यदि उपयोगकर्ता स्किल्स **नहीं** चाहता: `--skip-skills` जोड़ें

उदाहरण:

```bash
uvx --with playwright --from mfa-servicenow-mcp servicenow-mcp setup opencode \
  --instance-url "https://your-instance.service-now.com"
```

```bash
uvx --with playwright --from mfa-servicenow-mcp servicenow-mcp setup codex \
  --instance-url "https://your-instance.service-now.com" \
  --auth-type basic \
  --username "your-username" \
  --password "your-password"
```

```bash
uvx --with playwright --from mfa-servicenow-mcp servicenow-mcp setup claude-code \
  --instance-url "https://your-instance.service-now.com" \
  --scope global \
  --skip-skills
```

### Step 5 — इंस्टॉलेशन सत्यापित करें

1. पुष्टि करें कि इंस्टॉलर सफलतापूर्वक बाहर निकला
2. इंस्टॉलर सारांश द्वारा रिपोर्ट किया गया कॉन्फ़िग फ़ाइल पथ पढ़ें
3. यदि स्किल्स इंस्टॉल हुए थे, तो पुष्टि करें कि इंस्टॉल की गई स्किल्स डायरेक्टरी मौजूद है
4. कॉन्फ़िग को मैन्युअल रूप से पुनः न लिखें जब तक कि इंस्टॉलर विफल न हुआ हो और आप स्पष्ट रूप से उसकी मरम्मत न कर रहे हों

### Step 6 — उपयोगकर्ता को बताएँ कि आगे क्या होता है

इंस्टॉलेशन के बाद, उपयोगकर्ता को बताएँ:

> **इंस्टॉलेशन पूरा हो गया!**
> अपने AI क्लाइंट को पुनः आरंभ करें (या MCP सर्वर पुनः लोड करें) ताकि यह नए कॉन्फ़िगरेशन को उठा ले।
>
> पहले ब्राउज़र-प्रमाणित टूल कॉल पर, MFA/SSO लॉगिन के लिए एक ब्राउज़र विंडो खुलेगी।
> लॉगिन के बाद, आज़माएँ: `Run a health check on my ServiceNow instance`
> पूर्ण दस्तावेज़: https://jshsakura.github.io/mfa-servicenow-mcp/

सेटअप के बाद वर्तमान सत्र में ServiceNow MCP टूल्स को कॉल करने का प्रयास **न करें**। क्लाइंट को पहले पुनः आरंभ करना होगा।

### LLM के लिए महत्वपूर्ण नोट्स

- **कभी भी क्रेडेंशियल हार्डकोड न करें** बिना पूछे कॉन्फ़िग फ़ाइलों में। यदि उपयोगकर्ता क्रेडेंशियल छोड़ देता है, तो उन्हें कॉन्फ़िग से पूरी तरह छोड़ दें।
- इंस्टॉलर मौजूदा कॉन्फ़िग फ़ाइलों में मर्ज करता है। जब तक रिकवरी की आवश्यकता न हो, कॉन्फ़िग मर्ज को हाथ से न बनाएँ।
- **Windows पथ** बैकस्लैश का उपयोग करते हैं। OS के लिए सही पथ विभाजक का उपयोग करें।
- यदि कोई भी चरण विफल होता है, तो एरर का निदान करें और आगे बढ़ने से पहले उपयोगकर्ता को इसे ठीक करने में मदद करें।
- बातचीत को मित्रवत और संक्षिप्त रखें। टेक्स्ट की दीवारें न डालें।
- इंस्टॉलेशन के बाद, MCP टूल्स का परीक्षण करने की कोशिश न करें। बस उपयोगकर्ता को पुनः आरंभ करने के लिए कहें और वे तैयार हैं।
